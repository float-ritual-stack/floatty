# Dispatch Wiring

How verbs declared in a spec's `on:` map land on `useBlockStore`.

## The Flow

```
SPEC                             DOOR COMPONENT          HOST
────────────────────────────     ────────────────        ──────────────────────
{
  type: 'KanbanCard',         →  <Card on:drag={emit} →  chirp CustomEvent
  props: { … },                                           bubbles up
  on: {                                                   to inlineDoorRef
    drag: {                                               │
      action: 'move-block',                               ▼
      params: { … }                                       useDoorChirpListener
    }                                                     (registered in
  }                                                        BlockItem.tsx:226)
}                                                         │
                                                          ▼
                                                          chirpWriteHandler
                                                          switch case 'move-block'
                                                          │
                                                          ▼
                                                          store.moveBlock(…)
                                                          │
                                                          ▼ (Y.Doc transaction)
                                                          EventBus fires
                                                          │
                                                          ▼ (subscribed)
                                                          Door re-projects
```

## Where Each Piece Lives Today

| Piece | File | What it does |
|---|---|---|
| Verb declaration in spec | Generated by door spec generator (e.g. `kanbanSpec` in `doors/render/render.tsx:268`) | Emits `on` map on elements |
| Verb param schema | `doors/render/catalog.ts` `actions:` block | Zod shape |
| Door component `emit` | Any component in `doors/render/components.tsx` | Calls `ctx.emit('verb-name')` |
| Chirp dispatch bridge | `src/hooks/useDoorChirpListener.ts` | Listens for `chirp` CustomEvents on door root |
| Verb handler (write verbs) | `src/lib/chirpWriteHandler.ts` | Switch on verb name, call store method |
| Verb handler (nav verbs) | `src/components/BlockItem.tsx:226-245` `onNavigate` in the useDoorChirpListener deps | Call `handleChirpNavigate` |
| Store mutation | `src/hooks/useBlockStore.ts` | Surgical Y.Map / Y.Array ops |

## Adding a New Verb — Concrete Steps

Say you're adding `expand-block` for a tree view.

### 1. Zod schema in `catalog.ts`

```typescript
// doors/render/catalog.ts
actions: {
  // … existing …
  'expand-block': {
    params: z.object({
      blockId: z.string(),
      expanded: z.boolean(),
    }),
    description: 'Toggle a block\'s expanded state in a tree view.',
  },
},
```

### 2. Handler case in `chirpWriteHandler.ts`

```typescript
// src/lib/chirpWriteHandler.ts
case 'expand-block': {
  const { blockId, expanded } = data;
  if (!blockId) return { success: false };
  store.setBlockCollapsed(blockId, !expanded);  // note: inversion
  return { success: true, blockId };
}
```

Then extend `ChirpWriteStore` interface if `setBlockCollapsed` isn't
in there yet. Also add the verb to `isChirpWriteVerb()`.

### 3. Spec emits the verb

In the spec generator (e.g. `treeSpec` in `render.tsx`):

```typescript
elements[`node-${nodeId}`] = {
  type: 'TreeNode',
  props: { blockId: node.id, expanded: node.expanded },
  on: {
    toggle: { action: 'expand-block', params: { blockId: node.id, expanded: !node.expanded } }
  },
  children: [],
};
```

### 4. Component `emit`s, never handles

```tsx
// doors/render/components.tsx — TreeNode
export function TreeNode(props: BaseComponentProps<{...}>) {
  return (
    <div>
      <button onClick={() => props.emit('toggle')}>▸</button>
      {props.props.label}
    </div>
  );
}
```

Note: `props.emit('toggle')` dispatches the json-render action defined
in `on:`. The json-render `ActionProvider` resolves it to the
`action: 'expand-block'` entry, which bubbles as a chirp CustomEvent
with `message: 'expand-block', data: { blockId, expanded }`.

### 5. Grep-verify zero handlers in the door

```bash
grep -nE "onClick|onPointerDown|onDragStart|onDrop|addEventListener" \
  apps/floatty/doors/render/components.tsx | grep -v "emit("
```

If this returns lines that aren't `props.emit(...)` calls, the door is
still holding handlers it shouldn't.

## Why This Layering

- **Spec is declarative.** A future tool can walk the spec to learn what
  a view does without reading TypeScript. This matches ADR-002's
  "projections describe intent; canonical store executes."
- **Host has one dispatcher.** Changes to how `move-block` works — e.g.
  switching from pointer events to a newer drag-drop API — happen once,
  in the dispatcher. Every view benefits automatically.
- **Door has zero handlers.** Pure presentation. The same spec
  (`KanbanCard`, etc.) can be generated by a kanban spec generator OR
  by an AI-generated spec OR by a future config-driven spec. Interaction
  is the spec's responsibility, not the view's.

## When the Pattern Breaks Down

Some interactions genuinely require imperative code inside the
component — things like `elementFromPoint` during a pointer drag, or
caret positioning during edit. The escape hatch: those imperative
details stay inside the component, but the OUTCOME (the verb that
fires) must still go through `emit`. A pointer-drag component can
have window-level `pointermove` / `pointerup` listeners internally,
as long as the final decision ("this is a move-block from X to Y")
is routed through `emit('drag', {…})` and not through a bypass call
to `store.moveBlock`.

Rule of thumb: "I can have any amount of local component state and
computed values, but the moment I want to mutate outline state, I
emit a verb."
