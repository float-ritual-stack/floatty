## Parallel Systems (two things doing one job)

### Three Event Systems

Should be two (EventBus for block lifecycle, HookRegistry for execution middleware):

| System | Home | Used for | Status |
|---|---|---|---|
| `blockEventBus` | `src/lib/events/` | Block lifecycle — `outlinksHook`, `ctxRouterHook`, `funcRegistry`, `FilterBlockDisplay` | ✓ Correct home |
| `HookRegistry` | `src/lib/hooks/` | Execution middleware — `execute:before`, `execute:after` | ✓ Correct home, but see below |
| `ctxEvents.ts` | `src/lib/ctxEvents.ts` | ctx:: sidebar refresh signal | ✗ Should be an `blockEventBus` subscription |

**How `ctxEvents.ts` happened**: commit `e30f136` (Feb 7) — "replace sidebar polling with
event-driven refresh." Right diagnosis, right solution class. `blockEventBus` had been
available for 3+ weeks. A new 30-line DOM `EventTarget` was created instead of subscribing
to the existing bus filtered to `ctx::` prefix changes. `EVENTBUS_HOOK_MIGRATION_REVIEW.md`
even listed this as HIGH PRIORITY on the same day.

**The fix**: `ContextSidebar.tsx` subscribes to `blockEventBus` for `block:update` events
where content starts with `ctx::`. `Terminal.tsx` stops calling `emitCtxMarkersChanged`.
`ctxEvents.ts` deletes.

**`HookEvent` dead branch**: `types.ts` defines `HookEvent` as
`'block:create' | 'block:update' | 'block:delete' | 'execute:before' | 'execute:after'`.
`hookRegistry.run()` is only ever called with `execute:before` and `execute:after`.
The block lifecycle events were intended for HookRegistry per `FLOATTY_HOOK_SYSTEM.md`
but were implemented in EventBus instead. Either remove the three dead event types from
`HookEvent` or wire them. Currently the type says one thing, the code does another.

**Ticket**: FLO-423

### Two Navigation APIs

`useBacklinkNavigation.ts` is the original. `lib/navigation.ts` was added (Jan 13, `60a82f9`)
as a cleaner façade — its `navigateToPage` wraps `useBacklinkNavigation.navigateToPage`.

`BlockItem.tsx` currently imports both simultaneously with a rename:
```ts
import { navigateToPage, findTabIdByPaneId } from '../hooks/useBacklinkNavigation';
import { navigateToBlock, navigateToPage as navigateToPageNav } from '../lib/navigation';
```

**`findTabIdByPaneId`** is a pure layout utility with nothing to do with backlinks.
Imported from `useBacklinkNavigation` by 6+ files. Should live in `useLayoutStore` or `layoutTypes.ts`.

**The fix**: Retire `useBacklinkNavigation.navigateToPage` (make callers use `lib/navigation`
directly). Move `findTabIdByPaneId`. Don't break `findBacklinks`, `findPage`, `getPageTitle`
which are genuinely backlink concerns.

**Ticket**: FLO-424

## Presence Broadcast Spike Still Labeled Spike

`usePaneStore.setFocusedBlockId` contains:
```ts
// Spike: broadcast cursor position for TUI follower
if (blockId && window.__FLOATTY_SERVER_URL__) {
  fetch(`...`, { method: 'POST', ... }).catch(() => {});
}
```

Added Feb 15 (`aa12853`). The "Spike:" label means temporary/exploratory. It shipped
and has been in production for 3+ weeks. The correct home is an explicit `broadcastPresence()`
call at the call sites that care about it, not buried inside a store setter where it's
invisible. Store setters shouldn't have network side-effects.

**Ticket**: FLO-355 (Presence API protocol formalization) — already tracked.

## The RAF Situation

### Origin and Spread

The double-rAF pattern was invented in `a5d712e` (Jan 3) for `moveBlockUp`/`moveBlockDown`:

```ts
store.moveBlockUp(props.id);
// Double rAF: first for Y.Doc update, second for SolidJS DOM reconciliation
requestAnimationFrame(() => {
  requestAnimationFrame(() => contentRef?.focus());
});
```

This was **correct for that specific case**: moving a block reorders DOM nodes via
SolidJS's `For` component. The `contentRef` is temporarily between positions. Direct
`.focus()` during reconciliation is a no-op.

The pattern then spread to `create_block`, `zoom_in`, and navigation cases where it
doesn't apply. Y.js `transact()` fires observers **synchronously** before returning.
By the time `createBlockAfter()` returns, `state.blocks[newId]` already exists and
SolidJS has already reconciled. Calling `deps.onFocus(newId)` (which is `setFocusedBlockId`)
synchronously after would batch with the existing SolidJS render, the new BlockItem
would mount with `isFocused() === true`, and BlockItem's own single RAF in its
`createEffect` would handle the actual `.focus()`.

Current chain for new-block creation: **3 RAFs**. Should be: **1 RAF** (BlockItem's own).

### Taxonomy

| Location | Situation | RAFs needed | RAFs present |
|---|---|---|---|
| `moveBlockUp/Down` — `contentRef.focus()` direct | DOM reorder, same element, briefly detached | 2 | 2 ✓ |
| `createBlock` → `deps.onFocus(newId)` | New element, Y.Doc already settled, SolidJS already mounted | 1 | 3 ✗ |
| zoom wikilink → `deps.onFocus(targetId)` | Existing element, no reorder | 1 | 3 ✗ |
| `terminalManager` `fitAddon.fit()` scroll restore | xterm internal pipeline genuinely async | 2 | 2 ✓ |
| `OutlinerPane` initial scroll position | Reading DOM layout after mount | 1 | 1 ✓ |
| BlockItem `createEffect` for `isFocused()` | Waiting for browser to process focus after SolidJS render | 1 | 1 ✓ |

### How it contributed to keyboard regressions

The extra RAFs in `useBlockInput.ts` don't cause visible bugs directly — focus
happens eventually. But "add a RAF" became the established pattern for any timing
problem. The `onMouseDown` scroll preservation (added as a fix for scroll-jump on
block click) was exactly that instinct, and it created the FLO-278 race: two RAFs
both touching scroll position in overlapping windows. Fixed by removing `onMouseDown`
and using a CSS `.scroll-locked` class instead.

`domUtils.ts` was created to name the pattern (`afterDOMSettle`). It was never adopted
— the inline double-rAF continued to accumulate. The extraction was the deliverable,
not the adoption.

**Ticket**: FLO-425

## Keyboard Regression Root Causes (mostly fixed, documented for recurrence prevention)

The keyboard regressions were not caused by the RAF accumulation. They had three
separate sources:

### 1. Wrong contentEditable DOM model (biggest)

Three rounds of cursor fixes assumed `<div><br></div>` wrapping. floatty sets content
via `innerText` which creates **bare `<br>` at root**. Every cursor boundary function
was wrong for multi-line blocks. Affected: ArrowUp/Down not exiting blocks with
trailing newlines, Backspace incorrectly triggering merge, block split at wrong position.

**Status**: Fixed in `c3cbf56`. `cursorUtils.ts` rewritten with offset-based boundary
detection. 29 new tests. Rules documented in `.claude/rules/contenteditable-patterns.md`.

**Recurrence risk**: Any new cursor/navigation feature written without checking the
actual DOM first (via MCP `webview_execute_js`) re-enters the same cycle.

### 2. Content sync race (FLO-197)

150ms debounce + CRDT observer creates a window where typing in block A can be
overwritten when block B's update triggers SolidJS to re-run all content sync effects.

**Status**: Fixed with `hasLocalChanges` dirty flag. Flag set on input, cleared only
after debounced store commit completes.

### 3. Two RAF-based scroll mechanisms racing (FLO-278)

`onMouseDown` scroll preservation RAF + focus `createEffect` RAF both touched scroll
position in overlapping frames. Manifested as scroll stopping entirely after
zoom/paste/wikilink.

**Status**: Fixed. `onMouseDown` RAF removed. CSS `.scroll-locked` class used instead.
